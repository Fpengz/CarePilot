# Dietary Guardian Module Reference Documentation

## Overview

The `dietary_guardian` codebase is organized into a clean, layered architecture with the following structure:

- **Top-level**: Package initialization, versioning, and logging configuration
- **config/**: Application configuration management (app settings, LLM config, runtime settings)
- **core/**: Core primitives and domain-neutral contracts (IDs, types, errors, events, agent envelopes, notifications)
- **shared/**: Shared utilities (timezone/time helpers)

### `config/app.py`
**Purpose**: Composed application-level settings and environment bootstrap entry points

#### Classes

##### `AppIdentitySettings`
Defines core application metadata and image processing behavior.

**Fields**:
- `env: Literal["dev", "staging", "prod"]` — Deployment environment (default: "dev")
- `timezone: str` — Application timezone (default: "Asia/Singapore")
- `image_downscale_enabled: bool` — Enable image downscaling (default: False)
- `image_max_side_px: int` — Max image dimension in pixels (constraints: 256–4096, default: 1024)

**Config**: Uses `ConfigDict(populate_by_name=True)` for flexible field naming

---

##### `AppSettings`
Top-level composition of all application configuration subsystems.

**Fields**:
- `app: AppIdentitySettings` — App identity and metadata
- `api: APISettings` — API server settings
- `auth: AuthSettings` — Authentication and session management
- `channels: ChannelSettings` — Notification channel settings (Telegram, email, SMS)
- `emotion: EmotionSettings` — Emotion inference settings
- `llm: LLMSettings` — LLM provider selection and capability routing
- `observability: ObservabilitySettings` — Logging and observability settings
- `storage: StorageSettings` — Database and persistent/ephemeral storage backends
- `workers: WorkerSettings` — Background worker configuration

**Class Variables**:
- `_DEFAULT_SESSION_SECRET: ClassVar[str]` — Insecure dev default (must be overridden in prod)

**Key Methods**:

| Method | Signature | Purpose |
|--------|-----------|---------|
| `normalize_and_validate()` | `@model_validator(mode="after")` | Cross-group validation rules |
| `effective_google_api_key` | `@property` | Convenience accessor to LLM's effective Google API key |
| `from_environment()` | `@classmethod` | Bootstrap settings from environment variables |

**Validation Rules** (in `normalize_and_validate()`):
1. Sets `readiness_fail_on_warnings` based on environment (True for staging/prod)
2. Sets `seed_demo_users` based on environment (True for dev only)
3. Enforces `SESSION_SECRET` is never empty
4. For prod: enforces shadow tool policy → enforce and disallows demo users
5. For staging/prod: enforces session secret override, cookie security, demo user disabling
6. Enforces `COOKIE_SAMESITE=none` requires `COOKIE_SECURE=true`
7. Enforces Redis URL when `EPHEMERAL_STATE_BACKEND=redis`
8. Enforces Redis backend when `WORKER_MODE=external`

### `config/llm.py`
**Purpose**: Typed LLM provider, capability, and model-profile configuration contracts

Hierarchical configuration for LLM selection, per-capability routing, and provider-specific settings. All environment variables are read at the `LLMSettings` level for centralized pydantic-settings source resolution.

#### Enums

##### `ModelProvider`
**Type**: `StrEnum`

Available LLM providers:
- `GEMINI = "gemini"` — Google Gemini API
- `OPENAI = "openai"` — OpenAI API
- `OLLAMA = "ollama"` — Local Ollama server
- `VLLM = "vllm"` — Local vLLM server
- `CODEX = "codex"` — Reserved (not yet implemented)
- `TEST = "test"` — Testing mock provider

---

##### `LLMCapability`
**Type**: `StrEnum`

Distinct LLM capabilities used for per-capability routing:
- `CHATBOT = "chatbot"` — Conversational chat
- `MEAL_VISION = "meal_vision"` — Meal image analysis
- `DIETARY_REASONING = "dietary_reasoning"` — Dietary advice and reasoning
- `REPORT_PARSE = "report_parse"` — Parsing structured reports
- `CLINICAL_SUMMARY = "clinical_summary"` — Summarizing clinical data
- `FALLBACK = "fallback"` — Fallback capability for routing

---

#### Model Classes

##### `LLMCapabilityTarget`
**Base**: `BaseModel` (not `BaseSettings` — inline configuration, not env-read)

Per-capability routing override: provider, model, endpoint, and credentials.

**Fields**:
- `provider: ModelProvider` — Target provider
- `model: str | None` — Model name override (optional)
- `base_url: AnyHttpUrl | str | None` — API endpoint URL (optional)
- `api_key: str | None` — API key (optional)
- `api_key_env: str | None` — Environment variable name for API key (optional)

---

##### `LocalModelProfile`
**Base**: `BaseModel`

Named local model profile for Ollama/vLLM providers. Stored in `LLMSettings.local_profiles` and referenced by factory methods.

**Fields**:
- `id: str` — Profile identifier
- `provider: Literal["ollama", "vllm"]` — Provider type
- `model_name: str` — Model name (e.g., "qwen3-vl:4b")
- `base_url: str` — Service endpoint URL
- `api_key_env: str` — Environment variable for API key (default: "LOCAL_LLM_API_KEY")
- `enabled: bool` — Enable/disable profile (default: True)

---

##### `default_local_profiles()`
**Type**: Function → `dict[str, LocalModelProfile]`

Returns default local model profiles:
- `"ollama_qwen3-vl:4b"` — Qwen3-VL 4B Ollama profile (enabled by default)
- `"vllm_qwen"` — Qwen2.5 7B vLLM profile (enabled by default)

### `shared/time/timezone.py`
**Purpose**: Timezone conversion helpers reused by domain summary logic

#### Functions

##### `resolve_timezone()`
**Signature**: `(name: str) -> ZoneInfo`

Resolves a timezone name string to a `ZoneInfo` object. Falls back to UTC if name is invalid.

**Behavior**:
- Attempts to construct `ZoneInfo(name)`
- Catches `ZoneInfoNotFoundError` and returns `ZoneInfo("UTC")`

---

##### `local_date_for()`
**Signature**: `(dt: datetime, *, timezone_name: str) -> date`

Converts a UTC datetime to a local date in the specified timezone.

**Behavior**:
1. If `dt` is naive (no tzinfo), assumes UTC
2. Converts to target timezone using `dt.astimezone(resolve_timezone(timezone_name))`
3. Returns the `.date()` component

## ARCHITECTURAL SUMMARY

### Design Patterns

| Pattern | Usage |
|---------|-------|
| **Typed Config Objects** | `AppSettings`, `GeminiConfig`, etc. use Pydantic for validation and serialization |
| **Composed Settings** | `AppSettings` aggregates domain-specific settings (`APISettings`, `LLMSettings`, etc.) for unified configuration |
| **Structured Envelopes** | `AgentInputEnvelope`, `AgentOutputEnvelope`, etc. provide type-safe contracts for agent communication |
| **Protocol-based Ports** | `AlertRepositoryProtocol`, `ReminderNotificationRepository` define outbound contracts for persistence |
| **Value Objects** | `GeminiConfig`, `OpenAIConfig`, `LocalLLMConfig`, `InferenceConfig` are frozen dataclasses providing immutable, grouped config access |
| **Factory Functions** | `get_settings()`, `default_local_profiles()`, `build_api_error()` centralize object construction |

### Dependency Flow

```
config/
  ├── llm.py          (LLM routing, provider configs, model profiles)
  ├── runtime.py      (subsystem-specific settings)
  ├── app.py          (composition + cross-group validation)
  └── __init__.py     (public API)

core/
  ├── errors.py       (exception hierarchy)
  ├── events.py       (event envelopes)
  ├── ids.py          (ID types and generation)
  ├── types.py        (JSON type aliases)
  ├── contracts/
  │   ├── agent_envelopes.py  (agent I/O contracts)
  │   └── notifications.py     (notification repository ports)
  ├── config/__init__.py      (configuration exports)
  ├── time/__init__.py        (time utilities)
  └── __init__.py     (core primitives re-export)

shared/
  └── time/
      └── timezone.py  (timezone conversion utilities)
```

### Configuration Bootstrap Flow

1. **Environment Loading**: `.env` file loaded by `get_settings()` → `AppSettings.from_environment()`
2. **Subsystem Initialization**: Each subsystem (`APISettings`, `LLMSettings`, `StorageSettings`, etc.) reads its env-vars independently
3. **Composition**: `AppSettings` aggregates all subsystems
4. **Cross-Group Validation**: `AppSettings.normalize_and_validate()` enforces multi-subsystem business rules
5. **Caching**: Result cached via `@lru_cache(maxsize=1)`

### Key Architectural Concerns

1. **Centralized Environment Reading**: All env-var resolution happens at the leaf `BaseSettings` level, ensuring single source of truth
2. **Type Safety**: Pydantic validation + Python type hints ensure configuration is validated early and type-checked statically
3. **Modularity**: Subsystems are self-contained and independently composable
4. **Extensibility**: Agent envelope contracts support arbitrary metadata and policy flags for future extensions
5. **Auditability**: `AuditRecord`, `AgentExecutionTrace`, and `CaptureEnvelope` provide comprehensive request tracing
6. **Notification Decoupling**: Protocol-based repository ports allow flexible implementation (in-process, async workers, external services)
