## Architecture Decisions
- Decision: implement unified suggestions flow as a new API endpoint while preserving existing `reports/parse` and `recommendations/generate`
  - Why: additive change reduces frontend orchestration complexity without breaking existing API clients/tests
  - Tradeoff: temporary duplication remains until web flow fully migrates to suggestions endpoint
- Decision: persist suggestions in a dedicated `suggestion_records` table (instead of overloading `recommendation_records`)
  - Why: suggestions package report parse + recommendation + workflow + disclaimer and will support future listing/sharing use cases
  - Tradeoff: adds another table to legacy `SQLiteRepository` before full repository refactor
- Decision: create root compatibility docs (`ARCHITECTURE.md`, `SYSTEM_ROADMAP.md`, `API.md`) and `SAFETY.md`
  - Why: nightly automation prompt and external tooling expect these stable root files
  - Tradeoff: duplicate entrypoints must be kept in sync with canonical docs under `docs/`
