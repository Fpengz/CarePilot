## Architecture Decisions
- Decision: move suggestions orchestration into `application/suggestions/use_cases.py`
  - Why: keeps API transport thin and aligns with hexagonal migration
  - Tradeoff: temporary direct dependency on legacy repository/service modules until full repository port extraction
- Decision: keep `/reports` route as redirect compatibility shim
  - Why: preserve existing links while moving UX to unified suggestions flow
  - Tradeoff: additional compatibility route maintenance
- Decision: expose `scope=self|household` on suggestions list/detail
  - Why: enables household read-sharing using current membership model
  - Tradeoff: merged reads happen in use-case layer; dedicated read model can optimize later
- Decision: bind suggestions workflow ids to request context headers
  - Why: improves request/workflow traceability for debugging and auditability
  - Tradeoff: full structured logging/metrics pipeline is still future work
